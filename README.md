## Краткое описание задач

1. Основной функционал задачи №1 описан в `scripts/ozon_export.py`. Результат будет сохранен в `reports/`.
2. Telegram-бот (задача №2) описан в `bot/`. Задачи №1 и №2 логически объединены и имеют общую файловую структуру, задача №3 обособлена и расположена в своей директории.
3. Задача №3 описана в отдельной директории `cleaner/`. Там же находятся исходный файл, скрипт и полученный файл.

Реализовал таким образом, что запуск скрипта №1 тянет за собой уведомление о создании сводки в ТГ, при этом в самом боте также есть кнопка для формирования сводки.

Все команды для запуска скриптов в разделе "Запуск".

## Настройка

Создайте виртуальное окружение и установите зависимости:

```powershell
py -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

Заполните `.env` по примеру `.env.example`. 

## Запуск

Экспорт и отправка сводки (если заданы Telegram-настройки):

```powershell
python -m scripts.main
```

Запуск бота локально:

```powershell
python -m bot.main
```

Запуск скрипта очистки каталога:
```powershell
cd cleaner
python catalog_cleaner.py
```

## Docker

```powershell
docker compose up --build -d
```

Контейнер получает секреты из `.env`. Для планового утреннего запуска можно подтянуть любой планировщик.


## Отчет о выполнении задания

### Задачи №1 и №2 (Документация и ИИ)
При анализе документации `https://docs.ozon.ru/api/seller/` не пользовался ИИ, т.к. она достаточно качественная и есть встроенная консоль, пользуясь которой, я смог в браузере проверять работу эндпоинтов, смутило только то, что все эндпоинты исползуют метод `POST`. Когда я нашел два необходимых эндпоинта и проанализировал их, то создал .md файл, куда скопировал их описание из документации и сформулировал свой запрос, опираясь на документацю. Получив скрипт, самостоятельно провалидировал его, посмотрев, что эндпоинты те же и форматы запросов совпадают. На данном этапе могу еще отметить, что насторожило количество товаров у продавца - всего один товар с названием "Полотенце", но полагаю так и задумано, один из отчетов добавил в корень для примера `products_2026-07-21_14-52-23.csv`.

Также нашел библиотеку ozon-api, которая упрощает взаимодействие с Ozon Seller API, попробовал использовать её, но она сыровата, поэтому сгенерировал свой клиент для работы только с нужными в задании эндпоинтами API.

### Задача №3 (Крайние случаи и неоднозначные ситуации при очистке каталога)

- Сначала не разобрался, что такое OEM, перепутал с артикулом (`offer_id`). Позже логика была исправлена: OEM теперь извлекается только при явном указании в названии и помещается в отдельное поле.
- Для товара с `offer_id = MF1041` в разных строках встречались описания «Hyundai Solaris» и «Hyundai Solaris / Kia Rio». Посчитал это одной позицией для «Hyundai Solaris / Kia Rio».
- У товара `CF3300` цена была указана как «от 450 руб.», посчитал, что это просто `450`.
- Запись «Диск тормозной передний Mavico Solaris (без цены, уточнить)» не имеет цены и артикула. Сохранил её в таблице, т.к. она содержит важную информацию о наличии (6 шт.).
- Важно было заметить, что `offer_id` в названиях деталей мог писаться через пробел, дефис, либо слитно, и обработать эти случаи.
- Товар `CF3300` имеет в конце названия "пара", я посчитал, что в количество разумно записать 2 штуки.
